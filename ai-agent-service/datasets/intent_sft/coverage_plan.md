# 首期 300 条意图 SFT 覆盖计划

本计划用于扩充训练数据，避免只围绕少量固定句式堆叠样本。每个样本必须先进入人工复核队列，确认无敏感信息和无标签冲突后，才能分配到训练、验证或测试集。

## 分片规模

| 分片 | 数量 | 用途 |
| --- | ---: | --- |
| train | 210 | 供 QLoRA 学习业务语义与 JSON 输出格式 |
| validation | 45 | 选择 epoch、学习率和 LoRA 参数 |
| test | 45 | 只比较基线与微调版本，不参与训练和调参 |
| 合计 | 300 | 首期可运行的最小数据规模 |

## 业务覆盖配额

| 场景 | 数量 | 重点边界 |
| --- | ---: | --- |
| 退款：规则、步骤、进度、申请 | 65 | `policy_consult`、`how_to`、`status_query`、`action_request` |
| 物流与订单状态 | 40 | 物流查询、未收到、催单、订单号缺失 |
| 退货、换货、维修申请 | 50 | 动作申请与操作步骤的区分、槽位收集 |
| 发票与会员 | 30 | 开票动作、抬头/税号缺失、权益咨询 |
| 投诉、争议、赔付风险 | 45 | `need_human`、`need_ticket`、人工受控路由 |
| 人工转接与会话状态 | 25 | 明确转人工、工单进度、已有工单去重 |
| 无关与高风险越界问题 | 25 | 普通越界与医疗、法律、金融投资安全边界 |
| 安全与低置信度模糊表达 | 20 | 需要追问、意图不明、多业务词共现 |

## 防泄漏拆分规则

同一个表达簇只能进入一个分片。例如“退款多久到账”“退款几天到账”“退款多久退回”属于同一表达簇，不能分别放入 train 和 test。

每个场景至少保留 15% 的新措辞到 test。测试集中应有口语表达、省略主语、情绪表达和业务词共现的样本，避免模型只记住训练模板。

## 人工复核清单

每条候选样本在进入数据集前逐项确认：

1. 不含真实姓名、联系方式、地址、Token、完整订单号或工具原始返回。
2. `intent` 是业务域，`user_goal` 是用户目的，两者没有混用。
3. 真实动作、投诉、争议、人工转接均已正确标记人工与工单需求。
4. JSON 字段顺序、枚举值、空数组和 `null` 用法符合 `LLMIntentDraft`。
5. 没有与已发布黄金样本冲突的标签定义。
