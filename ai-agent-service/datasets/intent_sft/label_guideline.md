# 客服意图 SFT 标注规范

## 标注顺序

每条样本依次判断业务域 `intent`、用户目标 `user_goal`、人工与工单需求、以及 `next_action`。不要因为出现退款、换货或发票等词，就直接标注为真实动作申请。

## 用户目标边界

| 表达类型 | `user_goal` | 示例 |
| --- | --- | --- |
| 规则、条件、时效 | `policy_consult` | 退款通常多久到账？ |
| 操作入口或步骤 | `how_to` | 怎么申请退款？ |
| 当前个人订单或工单的真实进度 | `status_query` | 我的退款到哪一步了？ |
| 要求系统实际发起业务动作 | `action_request` | 帮我申请退款。 |
| 明确要求真人客服 | `human_request` | 请转人工客服。 |
| 投诉、维权、赔偿要求 | `complaint` 或 `dispute` | 少发商品，我要投诉。 |
| 无关的通用问题 | `out_of_scope` | 帮我推荐一只股票。 |

## 安全路由

投诉、争议、人工转接和真实业务动作请求必须标注 `need_human=true`、`need_ticket=true`。这只表示进入受控处理流程，不代表模型可以执行退款、赔付、取消订单或关闭投诉。

## 输出一致性

- 助手输出仅包含合法 JSON，不包含 Markdown 或解释。
- 字段顺序必须保持一致，遵循 `LLMIntentDraft` 的字段定义顺序。
- `confidence` 只使用 `0.95`（明确）、`0.75`（存在轻微歧义）或 `0.55`（低置信度）三档。
- 订单号只能使用 `EC_TEST_` 前缀的虚构标识；没有订单号时输出空数组。
- `next_action` 无后续动作时输出 `null`。
