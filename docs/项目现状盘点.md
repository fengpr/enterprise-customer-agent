## 2. 当前完整业务流程

### 2.1 客户提交售后争议并创建工单

典型输入：

```text
我买的鞋子质量不好，商家不给更换
```

调用链：

1. 客户侧 `frontend-demo/streamlit_app.py` 第 248 行点击“提交问题”。
2. 客户侧调用 `POST http://localhost:8000/api/agent/reply`，代码在第 254 行。
3. FastAPI 入口 `ai-agent-service/app.py` 第 59 行接收请求。
4. `app.py` 第 62 到 68 行调用 Java 校验 Token，并把可信 `customer_id` 和 `auth_token` 写入 Agent 请求。
5. `app.py` 第 75 行保存客户原始消息。
6. `app.py` 第 83 行调用 `CustomerServiceAgent.reply`。
7. `customer_service_agent.py` 第 58 行进入 Agent 门面。
8. LangGraph 在 `ticket_process_graph.py` 第 134 到 139 行按固定顺序执行：

```text
analyze -> query_order -> retrieve_knowledge -> risk_check -> create_ticket -> generate_reply
```

9. 售后争议识别：
   - `customer_service_agent.py` 第 148 行识别“质量不好/不给更换”为换货或售后争议。
   - 第 181 到 186 行将售后争议强制设为 `need_human=True`、`need_ticket=True`，并追加 `after_sale_dispute`。
   - 如果使用 LLM，第 409 到 416 行仍会通过业务兜底规则强制转人工和建单。
10. 建单节点：
    - `ticket_process_graph.py` 第 107 行进入 `create_ticket_node`。
    - 第 110 行判断 `need_ticket` 或 `need_human`。
    - 第 113 行组装工单 payload。
    - 第 114 行调用 `create_ticket`。
11. 工单 payload：
    - `ticket_process_graph.py` 第 144 行 `_build_ticket_payload`。
    - 关键字段包括 `title`、`ticketType`、`priority`、`customerId`、`orderNo`、`externalSessionNo`、`content`、`aiSummary`、`assignedGroup`、`source`。
    - `externalSessionNo` 保存 Python 会话编号，例如 `S202606...`，用于 Java 工单和 Python 会话强关联。
12. Python 工具调用 Java：
    - `ai-agent-service/tools/ticket_tools.py` 第 13 行 `create_ticket`。
    - 第 18 行调用 `POST http://localhost:8081/api/tickets`。
13. Java 创建工单：
    - `TicketController.java` 第 57 行接收 `POST /api/tickets`。
    - 第 62 行从 Token 解析客户 ID。
    - 第 63 行调用 `ticketService.createForCustomer`。
    - `TicketService.java` 第 81 行创建工单。
    - 第 87 行生成工单号。
    - 第 99 行默认状态为 `PENDING_ASSIGN`。
    - 第 100 行设置默认 24 小时 SLA。
14. Agent 返回客户安全话术：
    - `customer_service_agent.py` 第 113 行 `_build_customer_message`。
    - 客户只能看到“已受理、处理中、工单号、状态”等安全信息。
15. `app.py` 第 89 行保存 AI 回复消息，并在 `extra_data.ticket_result` 中保存 Java 返回的工单结果。

## 4. 关键边界和安全约束

1. Java `business-service` 是核心业务系统。Python Agent 不允许直接写核心业务表。
2. 工单创建、订单查询、工单状态流转必须通过 Java API。
3. 客户身份以 Java Token 为准。前端传入的 `customer_id` 不可信。
4. 客户侧只能展示客户安全话术、服务进度、工单号、工单状态和客户自己的会话。
5. 客户侧不得展示 `risk_reasons`、`decision_type`、AI 分析 JSON、工具调用结果、`internal_suggestion` 或审核按钮。
6. 坐席侧可以展示更完整上下文，但当前页面还没有完整展示 `risk_reasons`、`tool_results` 和 `internal_suggestion`。
7. Agent 可以辅助生成客户回复草稿，但不能自动发送。发送动作必须由坐席确认。
8. 高风险动作，例如退款、赔付、删除、关闭投诉等，不应由 Agent 自动执行。

- 坐席领取、开始处理、关闭工单。
- 客户侧刷新工单最新状态。
- 坐席填写处理结果。
- Agent 生成客户安全话术草稿。
- 坐席确认后发送客户回复。
- 客户侧刷新后看到“客服回复”。

### 5.2 Demo 特征

- Java 工单数据使用 SQLite 文件持久化，服务重启后可保留。
- Python 会话和 Agent 日志使用 SQLite 持久化。
- LLM 没配置或失败时，会回退到规则识别。
- Streamlit 没有实时推送，客户侧需要手动刷新。
- 角色权限是 Demo 账号级别，尚未接入真实权限系统。

## 6. 当前风险和技术债

### 6.1 工单和会话关联已初步强关联

已在 Java `SupportTicket` 中新增 `externalSessionNo` 字段，用于保存 Python 会话号，例如 `S202606...`。

当前新工单创建时，Python `_build_ticket_payload` 会写入 `externalSessionNo`，坐席回复草稿和确认发送会优先按该字段定位会话。

旧工单没有该字段时，系统仍会通过 `chat_message.extra_data.ticket_result.data.ticketNo` 反查会话，兼容代码在 `chat_repository.py:305`。

剩余风险：Java 工单和 Python 会话分别在两个 SQLite 文件中，当前还没有跨库事务；如果其中一侧写入失败，需要依靠接口错误和日志排查。

### 6.2 Java 工单已改为 SQLite，仍需补迁移体系

`TicketService` 当前通过 `JdbcTemplate` 写入 SQLite 文件 `business-service/data/business-service.db`。服务启动时会自动创建 `support_ticket` 表和常用索引。

建议：下一阶段补充正式迁移脚本或 Flyway/Liquibase，避免后续字段变更依赖代码内建表。

### 6.3 Java 编译验证环境不完整

当前工作环境没有 `mvn` 或 Maven Wrapper，之前 Java 编译未能执行。Python `compileall` 已通过，但 Java 需要在具备 Maven 的环境跑：

```bash
cd business-service
mvn -q -DskipTests compile
```

建议：补充 Maven Wrapper，避免环境差异。

### 6.4 坐席端上下文还不完整

当前坐席端展示了工单字段、客户问题、AI 摘要，但尚未完整展示：

- 会话历史
- `internal_suggestion`
- `risk_reasons`
- `tool_results`
- 客户上下文

这些信息已经保存在 Python 消息扩展字段中，但坐席端还没有完整读取和展示。

### 6.5 客户侧依赖手动刷新

Streamlit 没有实时消息推送。坐席发送消息后，客户必须点击“刷新最新回复和进度”或刷新页面。

建议：Demo 阶段可接受；正式阶段应做轮询、WebSocket 或 SSE。

### 6.6 坐席回复草稿当前是规则模板

当前 `/reply/draft` 的草稿由 `_build_staff_reply_draft` 生成，代码在 `ai-agent-service/app.py`，属于规则模板，不是真正调用 LLM 润色。

建议：后续接入 LLM，但必须保持“草稿生成，不自动发送”的边界。
