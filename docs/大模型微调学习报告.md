## 2. AI 应用开发中的能力边界

大模型应用并不等于“所有问题都靠微调”。企业系统中，需要将四类能力分开。

| 能力 | 解决的问题 | 客服项目中的例子 |
|---|---|---|
| Prompt（提示词） | 临时指令、输出风格与规则 | 要求回复简洁、先确认身份、输出 JSON |
| RAG（检索增强生成） | 会变化或大量的知识 | 退换货政策、运费规则、商品说明 |
| Tool Calling（工具调用） | 实时数据、受控业务动作 | 查询订单/物流、创建工单、获取工单状态 |
| Fine-tuning（微调） | 高频、稳定、可重复的行为模式 | 意图分类、槽位提取、风险路由、固定 JSON 契约 |

### 2.1 本项目适合微调什么

- 用户表达归一化：将“快递卡住了”“几天没动静”识别为 `logistics_query`；
- 结构化字段输出：如 `intent`、`need_order_query`、`need_ticket`、`need_human`、`next_action`；
- 稳定的风险路由：投诉、威胁曝光、赔付争议等场景建议转人工；
- 固定的机器可读输出格式，如严格 JSON。

### 2.2 本项目不应该由微调直接解决什么

- 实时物流状态、订单金额、工单状态；
- 随时可能变化的政策；
- 退款、赔付、关闭投诉等高风险操作权限。

原因是：训练数据只反映过去的样本，不能代替实时业务事实与权限控制。模型应识别“需要查询物流”，随后由 Tool 调用 Java 核心业务系统获取真实物流数据。Python Agent 不可绕过 Tool 直接写核心业务数据库。

## 3. Transformer 与 Self-Attention 基础

### 3.1 Token、Embedding 与上下文

- **Token（词元）**：模型处理文本的最小离散单位，不一定等于汉字、英文单词或字符。
- **Tokenizer（分词器）**：把文本转换为 token ID 的组件；不同模型的 token 划分不同。
- **Embedding（嵌入向量）**：token ID 查表后得到的稠密向量。模型处理的是向量，不是直接处理文字。
- **Context Window（上下文窗口）**：一次输入可容纳的 token 数上限。

### 3.2 Self-Attention（自注意力）

自注意力用于让当前位置结合上下文中其他位置的信息。每个 token 会变换出三组向量：

- **Q（Query，查询）**：当前位置想寻找什么信息；
- **K（Key，键）**：每个位置可被匹配的特征；
- **V（Value，值）**：匹配后实际聚合的内容。

核心计算可写为：

```text
Attention(Q, K, V) = softmax(QKᵀ / √d)V
```

其中：

- `QKᵀ` 表示查询和各键的相关性打分；
- `√d` 用于缩放，避免数值过大；
- `softmax` 把打分转成总和为 1 的注意力权重；
- 对 `V` 加权求和，得到当前位置吸收后的上下文表示。

### 3.3 Multi-Head Attention 与多层网络

- **Multi-Head Attention（多头注意力）**：多个注意力头并行关注不同关系，例如订单号与状态、否定词与投诉诉求、时间与物流事件。
- **多层 Attention**：底层倾向于组合局部词法信息，中间层更可能形成短语或语义关系，高层更接近任务与抽象意图表示。它不是手工规则，而是由训练数据和参数学习出的表示。
- **Causal Mask（因果掩码）**：生成式模型在预测当前位置时，不能查看未来 token，保证“只能根据历史预测下一个 token”。

对于 AI 应用开发，重点不是手推每层矩阵，而是理解：模型能从上下文建立关系，但实时事实、权限和业务流程仍必须交给系统工具与规则控制。

## 4. SFT：从黄金样本学习稳定行为

### 4.1 SFT 是什么

**SFT（Supervised Fine-Tuning，监督微调）**：用人工提供的“输入—理想输出”样本，继续训练预训练模型，使其更稳定地完成指定任务。

本项目的黄金样本采用聊天格式：

```json
{
  "messages": [
    {"role": "system", "content": "你是客服意图路由器，只输出 JSON。"},
    {"role": "user", "content": "物流一直没收到。"},
    {"role": "assistant", "content": "{\"intent\":\"logistics_query\", ...}"}
  ]
}
```

“黄金”不代表数量很大，而代表定义清楚、来源可靠、经过人工审核，并可作为训练或验收依据。

### 4.2 数据集切分

- **训练集（train）**：模型通过它更新参数；
- **验证集（validation）**：训练过程中观察泛化趋势、辅助选模型；
- **测试集（test）**：训练完成后才使用，用于模拟未知样本上的最终验收。

三者必须避免重复或近似泄漏。若把测试样本放进训练集，指标会虚高，无法代表真实线上效果。

### 4.3 数据质量比数量更优先

客服意图样本至少应覆盖：

- 多种表达：口语、错别字、简写、情绪化表达；
- 容易混淆的类别：政策咨询、订单查询、物流查询、售后、投诉；
- 槽位缺失：没有订单号、没有商品、没有明确诉求；
- 高风险情况：投诉升级、威胁曝光、赔付争议、敏感人群；
- 负例与边界例：看似“物流问题”但实际是地址变更或催单。

数据不能包含真实客户隐私。订单号、手机号、地址等必须使用脱敏或模拟标识。

## 5. LoRA 与 QLoRA

### 5.1 全参数微调的问题

预训练模型有大量原始参数，记为 `W`。全参数微调会更新 `W`，显存、存储、训练成本都很高，也更容易影响模型原有通用能力。

### 5.2 LoRA（Low-Rank Adaptation，低秩适配）

LoRA 冻结原模型权重 `W`，只训练两个小矩阵 `A` 和 `B`，近似一个低秩增量：

```text
W' = W + ΔW
ΔW = B × A
```

推理时使用带增量的 `W'`；训练时只更新 `A`、`B`。因此训练参数量和显存需求显著下降。

常见术语：

- **rank / r（秩）**：低秩矩阵的容量，越大通常表达能力越强、成本也越高；
- **alpha**：LoRA 增量的缩放系数；
- **target modules**：挂 LoRA 的目标层，例如注意力的 `q_proj`、`k_proj`、`v_proj`、`o_proj`，以及 MLP 的投影层；
- **Adapter（适配器）**：训练完成保存的 LoRA 小权重，不是完整基础模型。

本次训练输出显示：可训练参数约 2,993 万，基础模型总参数约 31.16 亿，训练比例约 **0.96%**。这就是 LoRA 降低成本的直观体现。

### 5.3 QLoRA（Quantized LoRA，量化低秩适配）

QLoRA 在 LoRA 的基础上，将基础模型以低比特形式加载，例如 4-bit NF4 量化；LoRA 适配器仍以较高精度训练。

关键概念：

- **Quantization（量化）**：用更少 bit 表示权重，减少显存；
- **4-bit**：每个量化权重近似使用 4 个二进制位表示；
- **NF4**：适合正态分布权重的一种 4-bit 量化格式；
- **double quantization（双重量化）**：进一步量化量化常数，继续节省显存；
- **bitsandbytes**：Python 生态中常用于低比特加载和训练的库。

对于 24 GB 显存的 RTX 4090，QLoRA 是学习和小规模项目验证的合理方案。

## 6. 训练脚本到底做了什么

训练主脚本位于 `ai-agent-service/training/train_intent_qlora.py`。它不直接实现每一个反向传播数学步骤，而是配置 Transformers、PEFT、PyTorch 这些框架，由框架执行训练循环。

### 6.1 数据如何进入模型

训练样本会经过 tokenizer，形成：

- `input_ids`：完整 token 序列，包含系统提示词、用户问题、助手标准答案；
- `attention_mask`：标识有效 token 与补齐 token；
- `labels`：每个位置用于计算训练误差的目标 token。

示意：

```text
input_ids:  用户问题 | <assistant> | { | "intent" | : | "logistics_query" | } | EOS
labels:     -100     | -100        | { | "intent" | : | "logistics_query" | } | EOS
```

其中 `-100` 是 PyTorch 交叉熵损失的默认忽略值，表示提示词部分不参与损失计算。模型不需要被训练成复述用户问题或系统提示词，而应学习生成助手答案。

### 6.2 标签右移（Shift Labels）

生成式模型的任务永远是：**根据前面的 token，预测下一个 token**。

因此框架内部对齐为：

```text
看到 <assistant>  → 预测 {
看到 {            → 预测 "intent"
看到 "intent"     → 预测 :
看到 :            → 预测 "logistics_query"
```

这被称为 **label shifting（标签右移）**：位置 `t` 的 logits 与位置 `t + 1` 的 label 比较。虽然 `{` 在答案区的第一个 label 位置，但它对应的是最后一个提示词 token 的预测结果。

### 6.3 EOS（End Of Sequence，序列结束标记）

训练脚本在每条助手答案后添加 `tokenizer.eos_token`。EOS 的作用是训练模型：完成合法 JSON 后应停止生成，而不是继续虚构下一段内容。

EOS 是自回归语言模型的通用机制，不是本项目特有。原始 JSONL 通常不需要手写可见的 `EOS` 文字；应使用当前模型 tokenizer 提供的结束 token。

### 6.4 Teacher Forcing 与实际推理

**Teacher Forcing（教师强制）**：训练时，模型生成下一个 token 所需的历史 token 来自正确标准答案。

```text
正确的 { → 预测 "intent"
正确的 "intent" → 预测 :
```

实际推理时没有正确答案，模型只能把自己刚生成的 token 接回上下文：

```text
用户问题 → 模型自己生成 { → 再基于 { 生成 "intent" → ...
```

这称为 **Autoregressive Generation（自回归生成）**。如果第一步生成错误，后续可能沿错误历史继续生成，这也解释了为什么训练 loss 下降不保证线上 JSON 一定合法。

## 7. Loss、梯度与参数更新

### 7.1 Logits 与 Softmax

模型在每一步会为整个词表的候选 token 输出原始分数，称为 **logits**。例如：

```text
{：8.2    我：6.1    您：5.7
```

**Softmax** 将 logits 转为总和为 1 的概率：

```text
{：80%    我：10%    您：6%    其他：4%
```

### 7.2 Cross-Entropy Loss（交叉熵损失）

对于正确 token，模型分配的概率越高，loss 越低。简化形式：

```text
loss = -log(p(正确 token))
```

训练时对所有未被 `-100` 忽略的助手答案 token 求平均。`eval_loss` 下降表示模型在验证集上更倾向预测标注中的下一个 token，但不直接等于业务正确率。

原因包括：

- 一个漏引号就能让 JSON 无法解析，但平均 token loss 可能仍不高；
- 每个字段的业务风险并不相同，漏转人工比一般字段错误严重得多；
- teacher forcing 下使用正确历史，而实际生成时模型要面对自身历史；
- 数据过少、覆盖不足或过拟合，都会造成泛化失败。

### 7.3 Backpropagation（反向传播）

基本训练循环可抽象为：

```text
前向计算：模型输出 logits 和 loss
反向传播：loss.backward() 计算各可训练参数的梯度
优化更新：optimizer.step() 用梯度更新 LoRA 参数
清空梯度：optimizer.zero_grad()
```

- **Gradient（梯度）**：loss 对参数变化方向与幅度的导数信息；
- **Learning Rate（学习率）**：每次更新走多大一步；太大不稳定，太小学习慢；
- **Batch Size（批大小）**：一次用于计算的样本数；
- **Epoch（轮次）**：完整遍历一次训练集；
- **Step（优化步）**：实际发生一次参数更新的次数；
- **Gradient Accumulation（梯度累积）**：连续多个小 batch 累积梯度后再更新，模拟较大 batch，节省显存。

本次初次训练只有 14 条训练样本，且使用 `gradient_accumulation_steps=16`、3 个 epoch，因此实际优化 step 极少。训练流程虽然跑通，但这不足以作为业务模型效果证明。

## 8. 解码策略与温度

微调改变的是 token 概率分布；**Decoding（解码）** 决定如何从该分布选出最终 token。

| 方法 | 规则 | 适用场景 |
|---|---|---|
| Greedy Decoding（贪心解码） | 总选最高概率 token | 意图 JSON、工具参数、风险路由 |
| Temperature Sampling（温度采样） | 调整概率分布后随机采样 | 自然语言回复、创意文案 |
| Top-k Sampling | 只从概率最高的 k 个候选中采样 | 可控多样性 |
| Top-p / Nucleus Sampling | 从累计概率达到 p 的候选集合中采样 | 更自然的开放式生成 |

### 8.1 温度的业务选择

- 对意图识别、风险路由、工具参数：`temperature=0` 或接近 0，追求稳定、可复现、易评测；
- 对客户安抚话术：可使用低温度如 `0.2~0.4`，使措辞更自然。

注意：低温度**不能保证业务判断正确**，它只减少随机性、稳定选择已有分布中概率最高的结果。正确性仍取决于数据、模型、评测和业务约束。

## 10. 离线评测：如何判断能否上线

离线评测脚本是 `ai-agent-service/training/evaluate_intent_adapter.py`。流程是：

```text
test.jsonl 人工黄金答案
  → 加载基础模型 + LoRA Adapter
  → 对每条用户问题贪心生成 JSON
  → 提取/校验 JSON
  → 与人工标注逐字段比较
  → 汇总指标并生成 adapter_evaluation.json
```

### 10.1 核心指标

| 指标 | 计算方式 | 回答的问题 |
|---|---|---|
| `schema_valid_rate` | 合法 JSON 数 / 总测试样本数 | 输出是否能被程序安全消费？ |
| `field_accuracy` | 某字段正确数 / 总测试样本数 | `intent`、`need_human` 等字段是否正确？ |
| `high_risk_route_recall` | 正确拦截的高风险样本 / 全部高风险样本 | 该转人工/建工单的风险请求是否被漏掉？ |

例如：

```text
schema_valid_rate = 100%
intent_accuracy = 92%
high_risk_route_recall = 55%
```

仍然不应直接上线自动处理所有请求。JSON 格式虽全合法，但 45% 的高风险请求可能没有被正确拦截，这是不可接受的业务风险。

### 10.2 当前评测结论

本次 5 条独立测试样本的初始结果中：

- 多个业务字段准确率为 0；
- `schema_valid_rate = 0`；
- `high_risk_route_recall = 0`。

因此当前 Adapter 不能接入生产路由。这是一个有价值的学习结果：训练完成不等于模型可用，必须用独立黄金集验证格式、语义与风险控制。

此外，当前评测脚本存在一个待修正的统计边界：当参考答案字段是 `null`、模型非法输出被记为 `{}` 时，使用 `prediction.get(field)` 可能把“字段缺失”误判成“正确的 None”。因此此前出现的 `next_action = 0.4` 不能视为有效成功率。后续必须修正后再比较实验。

## 11. 当前项目的安全落地原则

即使未来指标合格，微调模型也不应绕过系统边界。

```text
客户问题
  → Agent 鉴权、限流、入可靠队列
  → 微调模型输出意图/路由建议
  → Tool 调用 Java 业务服务获取实时订单/物流
  → 规则和权限控制高风险动作
  → 自动回复或创建工单/转人工
  → 客户侧仅展示安全话术与处理状态
```

关键原则：

- 微调输出是“受评测的建议/结构化判断”，不是绕过权限的最终业务事实；
- 高风险动作必须由 Java 业务系统权限与人工审核控制；
- 客户侧不可展示内部风险原因、Prompt、Trace、工具原始返回或分析 JSON；
- 线上请求走 Redis Stream 可靠队列与独立 Worker，不能把长时间 Agent 执行直接阻塞在 API 进程；
- 日志、Trace、队列、指标中不得保存 Authorization 原始 Token 或客户敏感信息。

## 12. 已掌握知识自测结论

已能够正确理解并回答：

- `input_ids` 包含提示词和助手标准答案；
- `labels` 的提示词部分使用 `-100` 忽略，助手答案部分才参与训练；
- 标签右移本质是“当前位置预测下一个 token”；
- 训练使用 teacher forcing，推理使用模型自身生成历史；
- logits 是候选 token 原始分数，softmax 将其转换为概率；
- 贪心解码选最高概率 token，适合结构化业务任务；
- 低温度提高确定性，但不能替代数据质量和业务校验；
- 合法 JSON 不代表业务判断正确；高风险召回率是上线关键指标之一。

## 13. 下一阶段学习与实践路线

### 阶段 A：完善最小实验闭环

### 阶段 B：数据工程能力

### 阶段 C：应用落地能力

3. 将实时订单/物流事实全部交给 Tool 调用；
4. 对高风险路由设置规则兜底与人工接管；

## 15. 一句话总复盘

> 微调不是让模型记住实时业务事实，而是用高质量黄金样本调整其 token 概率分布，使它更稳定地输出符合业务契约的意图与路由；真正的订单、物流与高风险动作仍由 Tool、权限、规则、队列和人工流程共同保障。
